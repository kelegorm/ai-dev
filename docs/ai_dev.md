# Разработка с помощью AI

Подход к работе с AI-агентами поверх Flutter-проектов. Архитектуру самих
проектов смотри в `arch.md`. Этот документ — про процесс, артефакты и
взаимодействие с агентами.

## Документация в проекте

В корне:

- `AGENTS.md` — главный агентский файл: гейт-команды, ссылки на скиллы,
  правила завершения задач.
- `CLAUDE.md` — `@AGENTS.md`.
- `AGENTS.codex.md` (или эквивалент) — `@AGENTS.md` для Codex, если тот не
  подхватывает корневой `AGENTS.md` сам. TBD: проверить.
- `roadmap.md` — короткий человеко-читаемый roadmap (см. ниже).

В `docs/`:

- `brainstorms/` — результаты CE-brainstorm.
- `ideas/` — сырые наброски до brainstorm.
- `plans/` — результаты CE-plan по фазам и фичам.
- `features/` — описание фич, на которые ссылается roadmap. Это «что входит
  в фичу», не план реализации.
- `architecture.md` — статичная карта проекта (слои, зависимости, правила).
  См. одноимённый раздел ниже.
- `arch/` — ADR (`INDEX.md` + `ADR-NNNN-*.md`) для решений, не очевидных из
  кода. Опционально, но желательно.
- `agents/skills/` или `.agent/skills/` (TBD) — агентские скиллы.

Подробный пример раскладки — в шаблонном репозитории (ссылка TBD).

## Архитектурный док

`docs/architecture.md` — один экран в длину. Не план, не roadmap, а
**карта**: слои, направление зависимостей, правила, чего нельзя.

- Итерируется **словами с пользователем**, не правкой кода. Три минуты
  диалога дешевле часа работы саба + проверка.
- Меняется до кода: правка едет в код только после обновления доки.
- Acceptance criteria внутри — конкретные грепы и команды, не проза.
  «Чисто» проверить нельзя; `grep -R "X" lib/ → пусто` можно.
- В брифе саба ссылается явно: «следуй буквально, противоречие —
  флагуй и предложи разрешение, не фантазируй».

## Enforcement

Что не enforce'ится — не существует. Два уровня:

- **Линтер.** `analysis_options.yaml` с правилами на `error`. Стандартный
  набор плюс плагин `dart_code_linter` с запретом `!`-оператора. Список
  правил эволюционирует — фиксируется в самом конфиге, с
  комментарием-почему рядом с каждым неочевидным.
- **Purity-тесты** в `test/architecture/`. Обычные dart-тесты, сканят
  `lib/**/*.dart`, фейлят `flutter test` при нарушении direction
  зависимостей или cross-screen импортов внутри `ui/`. Плагины-аналайзеры
  (`import_lint`, `custom_lint`) на текущем Flutter SDK не заводятся
  стабильно из-за конфликтов `meta` — тесты надёжнее и без отдельного
  тулинга.

Правило для саба, когда он ловит нарушение: **чинит код, не правило.**
В брифе явно: «не отключай, не понижай severity, не глуши через
`// ignore:`».

## Roadmap

Два уровня:

- **`roadmap.md` в корне** — для человека. Список фич в стиле «what's coming
  next», без деталей реализации. Сопоставимо с описанием на сайте продукта.
- **`docs/roadmap.md`** (детальный) — рабочий roadmap, который агенты
  заполняют в процессе. Каждая фича — компактный блок (≈10 строк, не больше
  20), `[x]` / `[ ]` для статуса.

Roadmap не содержит подробностей реализации. Каждая фича — ссылка на
`docs/features/<feature>.md` (что входит в фичу) и/или `docs/plans/<phase>.md`
(как будем строить).

## Процесс

Подход — **Compound Engineering** (CE), плагин `compound-engineering` для
Claude Code. Идея → roadmap → план → код через последовательность CE-скиллов
и сабагентов. Установка плагина — обязательное условие для всех шагов ниже.
Опционально полезен `superpowers` (пересекается частью практик: planning,
subagent-driven development, worktrees).

**Фазы:**

1. **Идея и стратегия.** Короткий набросок что строим и зачем. `ce-strategy`
   или вручную.
2. **CE-brainstorm.** Прокидываем основные фичи и развилки. На выходе —
   roadmap-набросок и список тем для углубления.
3. **Бутстрап проекта.** Из шаблонного репо генерим скелет (lib/, тесты,
   гейты, скиллы, AGENTS.md). Один-два простых экрана любого качества —
   главное, чтобы проект собирался и гейты были зелёные.
4. **Roadmap + features.** Заполняем `roadmap.md` и `docs/features/`.
   Уточняемся, пока roadmap не считается понятным.
5. **CE-plan.** На каждую фазу — `docs/plans/<phase>.md`. Plan дробит фазу
   на **batch'и задач**: каждый batch заканчивается компилируемым состоянием
   и помещается в контекстный бюджет одного сабагента.
6. **Plan-level review** (опционально). Дешевле code review, ловит
   архитектурные и сквозные косяки до того, как они станут кодом.
7. **Код.** Оркестратор-агент берёт план и раздаёт batch'и сабагентам.

## Оркестратор vs subagent

- **Оркестратор** держит контекст лёгким: читает план, формулирует
  компактные задания, проверяет результаты. Не лезет в реализацию, не
  перечитывает кучу файлов вместе с сабагентом.
- **Subagent** делает конкретный batch: пишет код, запускает
  `flutter analyze` / `flutter test` / DCL-гейты, возвращает короткий отчёт.
- Размер batch'а — столько, сколько помещается в контекст сабагента с
  запасом. Не лезет — дробить ещё.

**Зачем разделение.** Оркестратор остаётся отзывчивым: его можно прерывать,
спрашивать статус, добавлять детали. Если бы он сам делал работу, контекст
быстро бы засирался реализацией, и каждое «что там у тебя?» становилось бы
дорогим.

Конкретные правила поведения оркестратора — в `orchestrator.md`
(брифинг саба, проверка результата, типичные ошибки и реакция, общение
с пользователем). Подключается через `CLAUDE.md` проекта или
вставляется в системный промпт сессии.

## Rewrite, а не refactor

Если код агентно-сгенерён и нет mental ownership («что это за файл, не
помню, зачем»), двигание папок не помогает. Делаем rewrite **с
подсматриванием**:

1. `git mv lib/ lib_legacy/`.
2. Новый пустой `lib/` со скелетом слоёв и линтерами.
3. Перенос **по фиче**: каждая фича = коммит с зелёными гейтами. Из
   legacy копируем только pure-функции (математика, парсеры, чистая
   логика); плюминг (gateway'и, DI, транспорт) переписываем.
4. После всех фич — `git rm -r lib_legacy/`.

Замороженный контракт с бэком/внешним миром при этом не трогается.
Rewrite — про реализацию, не про спецификацию.

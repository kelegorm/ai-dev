## Зачем

В плагине `ai-dev` нужно оформить набор скиллов, которые помогают писать
Flutter-приложения в единой хорошей архитектуре и удерживать их в ней.
Цель — не объяснять агенту «мелочи жизни» заново для каждого нового
проекта, а сразу стартовать в ходовой архитектуре. Эталон уже описан
в `docs/reference-architecture/` и в репозитории-шаблоне `kelegorm/good_flutter_app`.

## Базовый принцип разделения

- **Эталон** (`docs/architecture/` плагина + шаблон `good_flutter_app`) —
  «как надо вообще». Живёт в плагине, меняется медленно и руками.
- **Истина конкретного проекта** — папка `docs/architecture/` **внутри
  проекта**. Самодостаточна: вся архитектура зафиксирована в проекте,
  без «дельты к эталону».
- Все скиллы после бутстрапа читают **проектные** доки, эталон не трогают.
  Это снимает конфликт «общая архитектура vs архитектура проекта».
- Скиллы и шаблон обновляются параллельно: если в проекте появился
  улучшенный enforcement-код — он вливается **в шаблон**, и `bootstrap`
  подтянет уже оттуда. Источник у `bootstrap` один.

## Организация плагина

- Скиллы лежат плоско в `skills/`, группировка несётся **префиксом
  имени**: `arch-bootstrap`, `arch-migrate`, `arch-audit`. Вызов —
  `/ai-dev:arch-<name>` (префикс `ai-dev` Claude Code даёт сам).
- Эталонная архитектура переехала в `docs/reference-architecture/`
  (отдельная папка, т.к. `ai-dev` шире архитектуры).
  Имя файла — со словом «эталон»/«reference», чтобы не путать с
  проектным.
- Каждый скилл — тонкий `SKILL.md` + папка `references/` рядом
  (progressive disclosure: тяжёлые шаблоны и техники грузятся по
  требованию).

### `arch-bootstrap`

- **Триггер:** явное намерение пользователя — «начинаю новый Flutter-
  проект», «разложи архитектуру», «настрой проект».
- **Область:** только greenfield (проекта нет / голый `flutter create`
  с пустым `lib/`). Проект с реальным кодом — это `arch-migrate`.
- **Результат:** проект разложен по эталонной архитектуре, в нём лежит
  набор арх-доков + `AGENTS.md`-индекс + работающий enforcement-код.

### `arch-migrate`

- **Триггер:** явный вызов на проекте, написанном «как попало».
- **Делает:** приводит существующий проект к архитектуре. Несколько
  видов рефакторинга: перекладка файлов по папкам, починка нарушений
  границ, выделение сервисов, выделение bloc из виджетов. Каждая
  техника — reference-док внутри скилла (`references/extract-bloc.md`
  и т.п.), не отдельный скилл.
- **Статус:** требует отдельного brainstorm. Здесь только зафиксирован
  факт его существования и область.

### `arch-audit`

- **Триггер:** явный вызов; плюс хук-гейт может подсказать «вызови
  audit».
- **Делает:** читает проектные доки по маркеру, сканит `lib/`, находит
  дрейф (файлы не в тех слоях, недостающие purity-тесты, дрейф линта),
  репортит и чинит.
- **Предусловие-гейт:** нет маркера в `docs/architecture/` → не аудитит,
  отправляет к `bootstrap`/`migrate`. Чужой `architecture.md` без
  маркера — не трогает.
- **Статус:** требует отдельного brainstorm.

### Хук-гейт

- Лежит в `hooks/hooks.json`, пути через `${CLAUDE_PLUGIN_ROOT}`.
- Гоняет **только детерминированные** проверки (гейт-команды:
  `flutter analyze`, purity-тесты, грепы). LLM-скилл из хука не
  запускается — хук может лишь заблокировать и подсказать вызвать скилл.
- Escapable: при провале блокирует, но понимает маркер обхода
  (например `wip:` в сообщении коммита) для намеренных WIP-коммитов.
- **Статус:** делается последним, детали — отдельно.

## `arch-bootstrap` — детальный дизайн

### Предусловия и шаги

0. **Проверка проекта.** Есть `pubspec.yaml` с `flutter:` + `lib/`?
   Нет → `flutter create` (как именно — скилл не документирует, агент
   умеет сам). Есть → шаг пропускается.
1. **Проверка маркера.** В `docs/architecture/` есть файл со штампом-
   маркером `ai-dev`? Да → проект уже инициализирован, не перезатираем,
   выходим. Нет → продолжаем.
2. **Скаффолд из шаблона.** `git clone --depth 1` репозитория
   `kelegorm/good_flutter_app`, копирование скелета слоёв и
   enforcement-кода, удаление `.git`. URL зашит в скилл. (Альтернатива —
   вендорить снапшот шаблона в плагин; решение — на этапе реализации.)
3. **Мини-диалог кастомизации.** Какие фичи, нужен ли auth-scope, нужна
   ли сеть — то, что делает `README.md` арх-доков проектно-специфичным.
4. **Генерация арх-доков** (см. ниже) со штампом-маркером в `README.md`.
5. **Генерация `AGENTS.md`** — живой индекс: ссылки на все арх-доки +
   `decisions/INDEX.md` + гейт-команды.
6. **Enforcement-код** — purity-тесты в `test/architecture/`,
   `analysis_options.yaml` — из шаблона.

### Набор арх-доков в проекте (`docs/architecture/`)

| Файл | Содержание |
|---|---|
| `README.md` | **Спина и индекс папки.** Заявление о стиле архитектуры (layered, смесь Clean + Hexagonal; разбивка по слоям, внутри `domain/` и `ui/` — по фичам; `domain/shared/` для cross-feature контроллеров/сервисов). Модель слоёв и направление зависимостей (таблица «кто что импортирует / никогда»). Аннотированное дерево папок: `app/` (composition root, DI), `app_ports/` (глобальные интерфейсы проекта — навигация и пр.), `domain/`, `ex_systems/`, `ui/`; где лежит файл роутера, где порт навигации. Стек. Ссылки на детальные файлы. Несёт штамп-маркер. |
| `tech.md` | DI и scopes, сеть (Dio + interceptors), механика навигации/роутинга (`AppNavigator`-порт, к auto_route напрямую никто не ходит), авторизация. |
| `ui.md` | Структура экрана, bloc/cubit, modal vs screen, навигация, design system, локализация. |
| `enforcement.md` | Доку-описание: чем проект механически защищён и как гонять — purity-тесты, конфиг линта, гейт-команды. Зубы — это код (шаг 6), а не этот файл. |
| `decisions/` | ADR (Architecture Decision Record): по файлу на неочевидное решение + `INDEX.md`. **Опционально** — `bootstrap` спрашивает при бутстрапе, создавать ли. |

Разрез — раскладка эталонного монолитного арх-дока (~440 строк) по темам, файлы
держатся небольшими (ориентир ≤ ~300 строк, плотный — повод дробить).

### Маркер

`bootstrap` штампует `README.md` арх-доков узнаваемым маркером
(строка-комментарий или frontmatter-поле вида `ai-dev arch contract
vN`). Все скиллы дальше опознают «наш» документ по маркеру, а не по
имени файла — старый проект с посторонним `architecture.md` нас не
путает.

## Отложено (не входит в эту итерацию)

- Подробный brainstorm `arch-migrate` и `arch-audit`.
- Детали хука.
- Вынос конвенций bloc/domain из эталона в отдельные reference-скиллы —
  когда `bootstrap` покажет, на что ему ссылаться.
- Скилл на заполнение/обновление `AGENTS.md` — принадлежит будущей
  не-архитектурной группе плагина.
- Возможный утилитарный скилл «выделить bloc» (skills-calling-skills) —
  только если техника подтвердится как часто-нужная отдельно.

## Подход к реализации

Скиллы пишутся по `superpowers:writing-skills` (TDD для скиллов:
baseline без скилла → скилл → проверка). Первым делается и тестируется
`arch-bootstrap` на сценарии «создать проект с нуля». Идеально с первого
раза не выйдет — пользователь тестирует на реальном проектике и правит
параллельно; спека — общий якорь.

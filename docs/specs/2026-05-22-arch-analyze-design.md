# Дизайн: `arch-analyze` — парадная дверь семейства

Статус: в работе. Эта итерация делает один скилл — `arch-analyze`.
Дата: 2026-05-22.
Опорные документы: `docs/specs/2026-05-21-flutter-arch-skills-design.md`
(семейство), `docs/specs/2026-05-22-arch-migrate-design.md` (миграция).

## Зачем

У семейства три рабочих входа — `arch-bootstrap` (greenfield),
`arch-migrate` (плоский бардак → миграция), `arch-audit` (дрейф
ai-dev-проекта). Но пользователь обычно **не знает**, какой из них
применим — он просто хочет «посмотри на проект, скажи что с ним, всё ли
хорошо». Под этот интент скилла нет: у `arch-migrate` описание узкое
(«мигрировать»), и попытка спросить «проверь архитектуру» ни на что не
триггерится.

Находка боевого теста `arch-migrate` (Task 7): пользователь вызвал
`arch-migrate` на уже разложенном-руками проекте — скилл вынужденно
сначала **анализировал** проект, чтобы понять, что мигрировать нечего.
То есть migrate-скилл делал работу несуществующего analyze-скилла, а его
таблица решений распухала строками-маршрутизаторами, которые ему не
принадлежат.

`arch-analyze` закрывает этот интент: осматривает проект, классифицирует
состояние, делает лёгкую оценку качества и рекомендует, что делать
дальше.

## Роль и место в семействе

- `arch-analyze` — **read-only парадная дверь**. Осматривает Flutter-
  проект, классифицирует состояние, выдаёт **отчёт + рекомендацию в
  чат**. Ничего не пишет (в т.ч. файла-отчёта — отчёт только в чате), не
  запускает другие скиллы, не чинит код. Это держит скилл безопасным —
  «вызывай когда угодно, хуже не сделает».
- Семейство **не перестраивается**: `arch-bootstrap` / `arch-migrate` /
  `arch-audit` остаются как есть, со своими предусловиями-гейтами (это
  защита при прямом вызове). `arch-analyze` — рекомендующий слой над
  ними, не оркестратор: он называет нужный скилл, но не диспатчит его.
- Триггер: analyze-интент — «проверь/оцени архитектуру», «что с
  проектом», «хорошая ли архитектура», «посмотри проект».

## Процедура

### 1. Survey — собрать факты

Без изменений в проекте:
- `flutter:` в `pubspec.yaml`? `lib/` пустой / плоский / слоистый?
- Имена папок верхнего уровня в `lib/`.
- Есть ли `test/architecture/` purity-тесты?
- Маркер `<!-- ai-dev:arch-contract v1 -->` первой строкой
  `docs/architecture/README.md`?
- Есть ли ledger `docs/architecture/migration-progress.md`?
- Есть ли собственный арх-док проекта.

### 2. Поиск признаков — пробы по сигналам

**Не полное сканирование кода** — пробы: несколько грепов плюс выборочно
заглянуть в пару файлов, чтобы понять «есть паттерн или нет». Детальный
чек-лист проб — в `references/signal-probes.md`. Пробы:

- **Паттерн представления.** Есть ли узнаваемый: bloc/cubit
  (`flutter_bloc`, `extends Bloc`/`Cubit`), контроллеры
  (`ChangeNotifier`, `*Controller`), MVVM-viewmodels, GetX, Riverpod —
  или ничего, логика прямо в виджетах. Вердикт: какой паттерн или «нет».
- **Смешение сущностей слоёв.** Свалены ли вместе (в одних файлах или
  папках) data-классы (DTO, `fromJson`), гейты/адаптеры и доменные
  модели. Проба: доменный файл с `fromJson` или с транспортным импортом;
  «одна папка на всё».
- **Навигация.** Есть ли выделенный навигатор (`AppNavigator`-порт,
  `auto_route`, `go_router`) — или повсюду ad-hoc `Navigator.push/pop`
  прямо в виджетах. Насколько чисто отделён (зовут ли экраны `Navigator`
  напрямую).
- **Слои и зависимости.** Папки-слои; направление импортов (греп
  межслойных «вверх», напр. `domain` → `ui`); есть ли purity-тесты.
- **Базовое.** `flutter analyze` чистый? явные запахи (data-fetch в
  `initState`, гигантские виджет-файлы)? имена слоёв vs эталон.

Каждая проба ограничена: несколько грепов + пара файлов. Read-only.

### 3. Классифицировать состояние

Отнести проект к одному из состояний (таблица ниже).

### 4. Отчёт в чат

Состояние + лёгкая оценка по пробам + рекомендация. Без файлов.

## Состояния → рекомендация

| Состояние | Рекомендация |
|---|---|
| greenfield — пусто или голый `flutter create` (`lib/` только сгенерированный `main.dart`) | Новый проект → `arch-bootstrap`. |
| плоский — реальный код, `lib/` без папок-слоёв | Бардак без слоёв → `arch-migrate`. |
| разложен по слоям, **нет** маркера ai-dev | Лёгкая оценка по пробам (паттерн / смешение / навигация / слои / запахи / имена vs эталон). «Архитектура есть, ai-dev ей не управляет.» Если раскладка отклоняется от эталона — честно назвать чем. Честно: привести разложенный-руками проект к эталону отдельным скиллом **пока нельзя** — вручную или будущая работа. |
| маркер есть, ledger с остатком | Миграция ai-dev в процессе; назвать остаток из ledger; `arch-migrate` продолжит, дальше — скиллы выделения. |
| маркер есть (ledger завершён, либо ledger'а нет — `bootstrap`) | Проект под управлением ai-dev. Для проверки дрейфа — `arch-audit` (ещё не написан — сказать прямо). |

## Что `arch-analyze` НЕ делает

- Не пишет файлов — отчёт только в чате.
- Не запускает другие скиллы (не оркестратор — рекомендатель).
- Не чинит код, не двигает файлы.
- Не делает глубокий разбор дрейфа — это `arch-audit`.

## Граница с `arch-audit`

| | `arch-analyze` | `arch-audit` |
|---|---|---|
| На каких проектах | любой (greenfield / плоский / разложен / маркер) | только с маркером ai-dev |
| Глубина | лёгкие пробы по сигналам | глубокий разбор дрейфа от зафиксированного контракта |
| Действие | только отчёт + рекомендация | репорт + **починка** |
| Для marked-проекта | отсылает к `arch-audit` | собственно делает разбор |

Дубля нет: для marked-проекта `analyze` не делает глубокий разбор, а
отсылает к `audit`. Лёгкую оценку `analyze` делает в т.ч. там, где
`audit` бессилен — на разложенном-руками проекте без маркера.

## Объём итерации

Один скилл `arch-analyze`: тонкий `SKILL.md` + `references/signal-probes.md`
(детальный чек-лист проб). Пишется по `superpowers:writing-skills` (TDD
для скиллов: baseline без скилла → скилл → проверка), как остальные.

## Отложено

- `arch-audit`, хук — свои итерации.
- Скилл, приводящий разложенный-руками не-эталонный проект к эталону —
  не существует; `arch-analyze` лишь честно сообщает об этом пробеле.
